# 抖医 TiktokRx 研究白皮书

## 基于多Agent辩论架构的抖音内容AI诊断系统

**版本**: 1.0
**日期**: 2026年5月

---

## 摘要

本白皮书介绍抖医 TiktokRx —— 一个基于多Agent辩论架构的AI抖音内容诊断系统。系统通过采集 8880+ 条真实抖音视频数据、6782 条评论、850 首热门 BGM，构建了数据驱动的基线知识图谱。在此基础上，6 个专业化 AI Agent 通过三轮辩论机制，对用户提交的抖音内容进行多维度量化诊断，输出包含六维评分、优化建议、评论区预测的综合报告。

---

## 1. 问题背景

抖音（Douyin/TikTok）作为全球领先的短视频平台，日均内容发布量超过数千万。然而，内容创作者面临以下痛点：

- **缺乏数据化反馈**：创作者只能通过播放量、点赞数等事后指标判断内容质量，无法在发布前获得诊断
- **优化建议笼统**：传统工具给出"提升内容吸引力"等泛泛建议，缺乏可执行性
- **单一视角偏差**：单次 AI 调用容易产生系统性偏差，缺乏多角度交叉验证
- **缺乏行业基线**：不同品类（美食、时尚、科技等）的内容标准差异显著，通用评分无意义

## 2. 系统架构

### 2.1 四阶段诊断引擎

```
Stage 1: 数据驱动基线训练
  ├─ 8880+ 条真实视频数据采集
  ├─ 6782 条评论数据标注
  ├─ 850 首 BGM 热度数据
  ├─ Spearman 相关分析
  ├─ 线性回归建模
  └─ K-Means 聚类 → 6 品类差异化基线

Stage 2: Model A 智能初评
  ├─ 纯数学模型，无 LLM 调用
  ├─ < 50ms 响应时间
  ├─ 6 维度即时打分
  └─ 品类差异化权重

Stage 3: 多智能体深度辩论
  ├─ Round 1: 6 Agent 并行诊断
  ├─ Round 2: 交叉质疑辩论
  └─ Round 3: 裁判综合裁定

Stage 4: AI 优化闭环
  ├─ 自动生成优化方案
  ├─ 即时重新评分
  └─ 最高分方案推荐
```

### 2.2 六维评分体系

| 维度 | 英文标识 | 评估内容 |
|------|----------|----------|
| 内容质量 | content_quality | 标题钩子、信息密度、文案结构 |
| 视觉表现 | visual_performance | 封面构图、色彩饱和度、文字占比 |
| BGM适配度 | bgm_adaptation | 音乐热度、节奏匹配、推流加持 |
| 增长策略 | growth_strategy | 标签策略、发布时间、流量入口 |
| 用户共鸣 | user_resonance | 评论区预测、情绪共鸣、互动引导 |
| 技术表现 | technical_performance | 画质清晰度、剪辑节奏、时长优化 |

## 3. 数据采集与处理

### 3.1 数据来源

所有数据通过公开渠道采集，仅用于研究目的。数据已脱敏处理，不包含任何个人隐私信息。

| 数据类型 | 数量 | 来源 | 说明 |
|----------|------|------|------|
| 视频数据 | 876 | 抖音公开页面 | 完整视频信息（标题、标签、互动数据） |
| 评论数据 | 6782 | 抖音公开评论 | 带类型标注（种草/经验/质疑/求助/凑热闹/吐槽） |
| BGM数据 | 850+ | QQ音乐多榜单 | 热歌/新歌/粤语/韩语/说唱/古典 |
| 标题数据 | 534 | 抖音热搜+合成 | 带钩子模式标注 |

### 3.2 统计方法

- **Spearman 秩相关分析**：识别与播放量/互动率显著相关的因素
- **线性回归**：建立内容特征与表现指标的量化关系
- **K-Means 聚类**：按品类分组，构建差异化基线
- **PCA 降维可视化**：验证聚类效果

## 4. 多Agent辩论系统

### 4.1 Agent 角色设计

| Agent | 角色 | 专注维度 |
|-------|------|----------|
| ContentAgent | 内容分析师 | 文案质量、信息密度、标题钩子 |
| VisualAgent | 视觉诊断师 | 封面构图、色彩、视觉吸引力 |
| BGMAgent | BGM策略师 | 音乐热度、节奏匹配、推流加持 |
| GrowthAgent | 增长策略师 | 标签策略、发布时间、流量入口 |
| UserSimAgent | 用户模拟器 | 评论区预测、用户情绪、互动引导 |
| JudgeAgent | 综合裁判 | 综合所有意见，输出最终报告 |

### 4.2 辩论机制

**Round 1 — 并行诊断**
6 位 Agent 同时独立分析内容，各自给出评分和诊断意见。每个 Agent 基于自己的专业领域和基线数据进行评估。

**Round 2 — 交叉辩论**
每位 Agent 读取其他人的意见，进行质疑、反驳或补充论据。这一阶段模拟了专家会诊中的交叉验证过程，有效减少单一视角偏差。

**Round 3 — 综合裁判**
裁判 Agent 综合所有意见（包括原始诊断和辩论中的新论据），给出最终评分、优化方案和评论区预测。

## 5. 评论画像引擎

基于 6782 条真实评论的分类训练，系统能够模拟 6 种用户画像的评论行为：

| 画像 | 占比 | 特征 |
|------|------|------|
| 种草型 (grass_type) | 30% | 推荐、安利、求链接 |
| 经验型 (experience_type) | 22% | 分享自身经历、对比 |
| 质疑型 (questioning_type) | 18% | 提出疑问、求证 |
| 凑热闹型 (crowding_type) | 15% | 围观、跟风、从众 |
| 求助型 (seeking_type) | 10% | 求教程、求方法 |
| 吐槽型 (help_type) | 5% | 吐槽、批评 |

---

## 6. BGM 热度模型

基于 850 首热门 BGM 数据，划分为 5 个热度等级：

| 等级 | 阈值 | 流量加持 |
|------|------|----------|
| S+ | 100万+ 播放 | +30% |
| S | 50-100万 | +20% |
| A | 10-50万 | +15% |
| B | 1-10万 | +5% |
| C | <1万 | 0% |

## 7. 技术实现

### 7.1 模型层级

| 层级 | 模型 | 用途 |
|------|------|------|
| 快速层 | MiMo-v2-Flash | 评论生成、快速任务 |
| 专业层 | MiMo-v2-Pro | Agent 诊断、辩论、裁判 |
| 多模态层 | MiMo-v2-Omni | 图像/视频分析、OCR |

### 7.2 系统架构

- **前端**: React 19 + TypeScript + MUI v9 + Vite
- **后端**: FastAPI + asyncio + SSE 流式推送 + SQLite
- **AI**: MiMo-v2 系列模型（OpenAI 兼容接口）
- **分析**: jieba 分词 + OpenCV + OCR + Whisper STT
