## 1. 文档说明与当前基线

### 1.1 当前已实现（基线）

- **前端**：React 19 + TypeScript + MUI v9 + Vite；首页（内容类型选择 + 文本/视频链接/图片上传）、诊断动画页（12 步 SSE 流式推送）、报告页（雷达图、Agent 折叠、模拟评论、卡片导出）；API 失败时使用 `frontend/src/utils/fallback.ts`。
- **后端**：FastAPI；6 Agent 多轮辩论（Content + Visual + Growth + UserSim + Judge + BGM）；`POST /api/diagnose`（multipart，支持视频/图片/纯文本）、`POST /api/generate-comments`、`GET /api/baseline/{category}`、`POST /api/analyze-bgm`、`POST /api/analyze-text`；Baseline 对比（SQLite `tiktok_baseline.db`）。
- **数据**：`scripts/compute_baseline.py` 计算基线；`backend/data/tiktok_baseline.db`（含 bgm_database 850 条、title_database 534 条、comment_database 2502 条）。
- **Agent**：`base_agent.py` 封装 MiMo 网关适配（`max_completion_tokens`、`trust_env=False`）；Model 三档：PRO（诊断/辩论/裁判）、OMNI（多模态视觉）、FLASH（评论生成/快速任务）。

### 1.2 本文档定位

- **不是**对已实现功能的重复说明；**是**对白皮书 [`docs/whitepaper.md`](../docs/whitepaper.md)、产品与路演叙事的 **差距清单** 与 **可执行任务书**。
- 每项尽量包含：**目标**、**现状**、**要做事项**、**验收标准**、**主要改动文件/模块**、**风险与备注**。

## 2. 产品与体验缺口

### 2.1 抖音视频链接导入（P0）

| 项目 | 内容 |
|------|------|
| **目标** | 用户粘贴抖音分享链接，系统自动解析出视频信息（标题、作者、封面 URL），减少手动输入。 |
| **现状** | `backend/app/api/diagnose.py` 已支持视频链接输入，`POST /api/parse-link` 存在解析接口；但解析成功率与容错策略待完善。 |
| **要做事项** | ① `frontend/src/pages/Home.tsx` 增加链接输入自动校验与预览（封面缩略图 + 标题）；② 后端 `parse-link` 增加重试/多格式兼容（短链、标准分享格式）；③ 失败策略：提示用户改用截图 OCR 或手动填写。 |
| **验收标准** | 至少一种常见分享格式可稳定走完「链接 → 预览 → 一键诊断」；失败时有明确文案，不白屏。 |
| **主要文件** | `frontend/src/pages/Home.tsx`、`frontend/src/utils/api.ts`、`backend/app/api/diagnose.py`。 |
| **风险** | 平台反爬、协议变更；答辩时准备「合规说明」。 |

### 2.2 截图 OCR 接入主流程（P0）

| 项目 | 内容 |
|------|------|
| **目标** | 用户上传抖音截图时，系统自动抽取标题、正文、标签（及可见信息），减少手工输入。 |
| **现状** | `backend/app/analysis/ocr_processor.py` 具备基于视觉 LLM 的思路；`backend/app/api/screenshot_api.py` 有独立接口，但 **主诊断路径未调用** OCR 预填充。 |
| **要做事项** | ① 在 `orchestrator.py` 或 `diagnose` 入口：若 `cover_image` 存在且 `title/content` 为空，先调 OCR 预填；② 合并/覆盖策略（OCR 结果与用户手改冲突时以用户为准）；③ 成本控制：可选「仅 OCR 预览」模式不跑全量 Agent。 |
| **验收标准** | 纯截图路径可输出标题 + 部分正文作为预填；日志中可追踪 OCR 成功/失败。 |
| **主要文件** | `backend/app/api/diagnose.py`、`backend/app/agents/orchestrator.py`、`backend/app/analysis/ocr_processor.py`、`backend/app/api/screenshot_api.py`。 |
| **风险** | API 费用与延迟；需 fallback。 |

### 2.4 诊断进度与真实阶段同步（P1）

| 项目 | 内容 |
|------|------|
| **目标** | 动画/进度与后端实际阶段一致，避免「已跳到最后一步但接口仍在等 LLM」。 |
| **现状** | `Diagnosing.tsx` 使用 SSE 流式推送 12 步，但后端 `orchestrator.py` 各阶段耗时差异大（6 Agent 并行 vs 裁判串行）。 |
| **要做事项** | ① 后端 `orchestrator.py` 分段 yield SSE event（阶段名 + 预估耗时）；② 前端根据 SSE event 实时推进步骤，而非固定间隔；③ 弱网降级：超时后展示「正在生成报告...」而非静止。 |
| **验收标准** | 网络慢时不出现「步骤全绿但长时间不跳转」；步骤与后端日志阶段可一一对应。 |
| **主要文件** | `frontend/src/pages/Diagnosing.tsx`、`backend/app/api/diagnose.py`、`backend/app/agents/orchestrator.py`。 |

| **验收标准** | 答辩可演示「原文 → 优化标题 + 改写文案 + BGM 推荐 + 封面 checklist → 新评分」。 |
| **主要文件** | `backend/app/api/optimize_api.py`、`backend/app/agents/orchestrator.py`、`backend/app/agents/prompts/judge_agent.py`、`frontend/src/pages/Report.tsx`。 |
